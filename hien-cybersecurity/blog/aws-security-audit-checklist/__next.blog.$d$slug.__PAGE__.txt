3:T17f5,<h1>AWS Security Audit Checklist: What Auditors Check and How to Prepare</h1>
<p>Most AWS audit failures are predictable: public data exposure, weak identity boundaries, missing logging, and untracked infrastructure drift. This checklist is for CTOs and platform leads who need to prepare for an external AWS security audit or SOC 2 without turning engineering into a compliance bottleneck.</p>
<hr>
<h2>What an AWS security audit usually covers</h2>
<p>An effective audit aligns with a known framework (e.g. CIS AWS Foundations Benchmark) and focuses on identity, data, network, and evidence. The sections below map to what auditors typically check and what you can fix in advance.</p>
<hr>
<h2>1. IAM and identity</h2>
<h3>1.1 Root and MFA</h3>
<ul>
<li>Root account has MFA and is not used for day-to-day operations.</li>
<li>No long-lived access keys for human users; use SSO or IAM Identity Center with short-lived credentials.</li>
<li>Privileged roles (admin, power user) require MFA.</li>
</ul>
<h3>1.2 Least privilege and escalation paths</h3>
<ul>
<li>No wildcard (<code>*</code>) permissions except where strictly required and documented.</li>
<li>Roles are scoped to environment (dev/stage/prod) and purpose.</li>
<li>There are no privilege-escalation paths (e.g. a role that can attach a new policy granting itself admin).</li>
<li>Unused users, roles, and access keys are removed or disabled.</li>
</ul>
<h3>1.3 Cross-account and assume-role</h3>
<ul>
<li>Cross-account access is documented and uses least-privilege role assumption.</li>
<li>No hardcoded long-lived keys for cross-account access where assume-role is possible.</li>
</ul>
<p><strong>Internal link:</strong> For a structured <a href="../../services/aws-security-audit/">AWS Security Audit</a> (CIS-aligned review of IAM, S3, network, and logging), see the service page.</p>
<hr>
<h2>2. S3 and storage</h2>
<h3>2.1 Public access</h3>
<ul>
<li>Block Public Access is enabled at account and bucket level.</li>
<li>No bucket policies or ACLs grant public read/write/list unless there is a justified, documented exception.</li>
</ul>
<h3>2.2 Encryption and transport</h3>
<ul>
<li>Default encryption is enabled (SSE-S3 or SSE-KMS).</li>
<li>Bucket policies deny non-TLS requests where applicable.</li>
<li>Sensitive data is not stored in buckets that allow public or overly broad access.</li>
</ul>
<h3>2.3 Lifecycle and classification</h3>
<ul>
<li>Retention and lifecycle rules exist where required for compliance.</li>
<li>There is a simple data classification approach so teams know what can live in which buckets.</li>
</ul>
<hr>
<h2>3. Network and security groups</h2>
<h3>3.1 Inbound exposure</h3>
<ul>
<li>No security groups allow <code>0.0.0.0/0</code> on SSH (22) or RDP (3389); use SSM Session Manager, bastions, or VPN with strict access.</li>
<li>Production databases and internal APIs are not open to the internet.</li>
<li>Any rule allowing 0.0.0.0/0 is documented and reviewed.</li>
</ul>
<h3>3.2 Segmentation and flow logs</h3>
<ul>
<li>VPC and subnet design support segmentation (e.g. web / app / data tiers).</li>
<li>Flow logs are enabled for critical VPCs or subnets.</li>
<li>Network ACLs are used where they add defence-in-depth.</li>
</ul>
<hr>
<h2>4. Logging and monitoring</h2>
<h3>4.1 CloudTrail</h3>
<ul>
<li>CloudTrail is enabled in all regions you use (or organisation-wide).</li>
<li>Logs are centralised in a dedicated account or bucket with restricted access.</li>
<li>Log integrity (e.g. log file validation) is enabled.</li>
<li>Retention meets compliance requirements.</li>
</ul>
<h3>4.2 Config and GuardDuty</h3>
<ul>
<li>AWS Config is enabled where required for compliance and drift detection.</li>
<li>GuardDuty (or equivalent) is enabled and findings are reviewed.</li>
<li>There is a process to respond to critical findings.</li>
</ul>
<h3>4.3 Alerting</h3>
<ul>
<li>Alerts exist for sensitive actions: IAM policy changes, new access keys, disabling of logging, public S3 policy changes, and similar.</li>
<li>Ownership and response procedures for alerts are defined.</li>
</ul>
<hr>
<h2>5. Secrets and configuration</h2>
<ul>
<li>No long-lived secrets in plaintext in user data, environment variables, or public parameter stores.</li>
<li>Secrets are stored in Secrets Manager or Parameter Store (encrypted) and access is restricted by IAM.</li>
<li>CI/CD and automation do not log secrets; secret scanning is in place and blocks merges when leaks are found.</li>
</ul>
<hr>
<h2>6. Infrastructure as code and drift</h2>
<ul>
<li>Critical resources are managed as code (e.g. Terraform, CloudFormation).</li>
<li>There is a security or policy check in the pipeline (e.g. policy-as-code, IaC scanning).</li>
<li>Drift between code and live resources is detected and reviewed (e.g. weekly).</li>
</ul>
<hr>
<h2>A practical 2-week audit readiness plan</h2>
<table>
<thead>
<tr>
<th>Phase</th>
<th>Focus</th>
<th>Actions</th>
</tr>
</thead>
<tbody><tr>
<td><strong>Days 1–2</strong></td>
<td>Identity + logging</td>
<td>MFA everywhere; remove unused IAM; enable CloudTrail in all regions; centralise logs and set retention.</td>
</tr>
<tr>
<td><strong>Days 3–5</strong></td>
<td>Data + network</td>
<td>Block public S3; default encryption; tighten security groups; remove 0.0.0.0/0 on SSH/RDP.</td>
</tr>
<tr>
<td><strong>Week 2</strong></td>
<td>Guardrails</td>
<td>IaC security checks in CI; drift detection; define vulnerability/patch SLA and tracking.</td>
</tr>
</tbody></table>
<p>This gets most teams from “audit panic” to a defensible baseline without heavy process.</p>
<hr>
<h2>What you should get from an AWS audit</h2>
<p>A useful engagement should deliver:</p>
<ul>
<li>A prioritised list of findings with risk ratings and affected resources.</li>
<li>Clear, implementable fixes that engineers can execute.</li>
<li>Recommendations for guardrails so the same issues do not reappear.</li>
<li>Reporting suitable for leadership and auditors.</li>
</ul>
